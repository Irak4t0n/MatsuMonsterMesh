## 1. Summary

The port is in good shape: the daycare social layer, Gen 1/2 party reading, LORD
campaign, channel system, chat UI, and MQTT bridge are solid and largely match
upstream semantics. The review found **one leaked credential, four real bugs, two
wire-format parity gaps (both fixed), and four major upstream features that are
not ported** (documented below with implementation notes).

Everything marked ✅ FIXED was changed in this session and needs a
`idf.py build` + hardware test before pushing.

### Recommended porting order (if you want full parity)
1. **`hb` + `news` commands** — trivial, immediate UX win.
2. **Networked PvP** — port the 0x66–0x6C server-authoritative protocol from
   `BattlePacket.h`/`MonsterMeshTextBattle.{h,cpp}` upstream rather than wiring
   the legacy lockstep initiator: it's what current T-Decks speak, it tolerates
   lossy links (resend + piggyback ACK + 24-bit board hash + full-state repair),
   and only the server runs the engine (no determinism requirement). Keep the
   existing lockstep receiver for back-compat.
3. **Gauntlet** — self-contained; needs directed text DMs (currently
   `daycare_send_dm_cb` broadcasts instead of DMing — fix that first).
4. **Dungeon** — largest effort, save for last.

## 4. Additional observations (no action taken)

- **MQTT-only MonsterMesh channel**: intentional per Session 13, but it means a
  LoRa-only unit (no WiFi) never transmits daycare/battle traffic. If you ever
  want offline park-bench play vs a T-Deck, add a LoRa fallback in
  `send_data_frame_mm` (the T-Deck listens on LoRa channel 1).
- **DM path is broadcast**: `daycare_send_dm_cb` broadcasts flavor text to
  everyone instead of DMing the partner node. Fun, but noisy on shared
  channels, and Gauntlet will need real directed DMs.
- **`MeshtasticTransport.h`** in monster_core is entirely `#if 0` — reference
  only; delete when no longer useful.
- **Timing divergences from upstream** (fine, just know): event interval 30 min
  here vs 5 min upstream-code/30 min upstream-docs; neighbor timeout 2 h (same);
  beacon 5 min here vs 15 min upstream.
- **`s_ngPlusTier`** in LordLogic.cpp is a process-global set by the terminal;
  beacon TX now reads it — make sure `lordSetCurrentNgPlusTier()` is called on
  LORD save load (it is, via terminal init path) so the first beacon after boot
  carries the right tier.
- Wire-parse safety generally good: beacon RX clamps to `sizeof`, party-chunk
  reassembly bounds-checked, TLS cert verification enabled (`esp_crt_bundle`),
  WiFi creds properly in NVS.
